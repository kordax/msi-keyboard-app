# Протокол MSI Strike Pro

Это рабочие заметки по приёмнику `0db0:1620` и проводному устройству
`0db0:b231`. Они описывают только то, что подтверждено дескрипторами и
трассами подключённой клавиатуры. Неизвестные команды приложение не
отправляет.

## USB и HID-интерфейсы

Снимок от 19 июля 2026 года:

| PID | Interface | hidraw | Назначение по HID report descriptor |
|-----|-----------|--------|--------------------------------------|
| `1620`, `b231` | 0 | динамический | Обычная клавиатура, boot protocol |
| `1620`, `b231` | 1 | динамический | Vendor page `0xff00`: input 64 байта и output 64 байта, без report ID |
| `1620`, `b231` | 2 | динамический | Feature `0x07` на 63 байта, feature `0x0c` на 10 байт, мышь и consumer control |
| `b231` | 3 | динамический | Дополнительный HID descriptor на 327 байт, назначение ещё не декодировано |
| `1620`, `b231` | 4 | динамический | Vendor page `0xff0d`: input report `0x0d` на 63 байта |

Пути `/dev/hidrawN` меняются между подключениями. Код выбирает интерфейс по
VID, PID и номеру USB-интерфейса, а не по `N`.

## Безопасные операции

Сейчас реализованы только:

- пассивное чтение input report на интерфейсах 1 и 4;
- `GET_FEATURE` для объявленных дескриптором report `0x07` и `0x0c`;
- `GET_INPUT` для vendor-report `0x00`, `0x06` и `0x0d`;
- `GET_OUTPUT` для чтения текущего состояния vendor-report `0x00`;
- точный 64-байтовый запрос батареи, подтверждённый трассами MSI Center;
- экспорт сырых пакетов без USB serial number;
- декодирование процента только из ответа с полностью совпавшим префиксом.

`SET_FEATURE`, произвольные USB control transfer и неизвестные output-команды
не используются. `GET_OUTPUT` является чтением состояния, а не отправкой
output report.

## Первый снимок с USB-приёмника

Снимок от 19 июля 2026 года после корректной выдачи `uaccess`:

- `GET_FEATURE 0x07` вернул 64 байта: `0x07` и 63 нулевых байта;
- `GET_FEATURE 0x0c` вернул 11 нулевых байт, включая `0x00` вместо
  запрошенного report ID;
- за 8 секунд пассивного наблюдения интерфейсы 1 и 4 не выдали input-report.

CLI хранит запрошенный feature report ID отдельно от первого байта ответа.
Это важно для `0x0c`, где firmware затирает или не возвращает report ID.

Нулевые ответы не являются значением батареи. Текущие данные указывают, что
заряд либо выдаётся только в ответ на отдельную vendor-команду, либо
публикуется при событии, которое ещё не воспроизведено.

Дополнительный read-only снимок запросил текущее состояние через `GET_INPUT`
и `GET_OUTPUT`. Vendor input `0x0d` и output `0x00` также вернули только нули,
а input `0x00` и `0x06` не вернули данных. Эти запросы не изменяют устройство,
но заряд без предварительной vendor-команды не раскрывают.

## Подтверждённый запрос батареи

Три VirtualBox-трассы от 19 июля 2026 года содержат десять одинаковых запросов
MSI Center на interface 1 в беспроводном и проводном режимах. USB payload
состоит из 64 байт:

```text
0d b0 01 00 00 00 05 00 00 00 [ещё 54 нулевых байта]
```

Ответ начинается с `0d b0 01 00 00 00 05 02`. Процент находится в байте 8,
если считать первый `0d` байтом 0. Во второй трассе MSI Center показывал 78%,
а три ответа battery-query содержали 78, 78 и 79. Первая трасса содержала
значения 85–87; первоначальная оценка «около 94%» оказалась неточной.

Интерфейс не использует HID report ID. Для Linux `hidraw write()` перед
64-байтовым payload добавляется нулевой report ID, поэтому размер буфера
составляет 65 байт. Код отправляет только этот подтверждённый запрос.

Первый независимый запуск `--cli --battery` в Linux вернул 76% примерно через
одну миллисекунду после запроса. За несколько минут до этого MSI Center
показывал 78%, что подтверждает как поле процента, так и его изменение при
работе клавиатуры.

Проводная трасса `0db0:b231` снята при активной зарядке. MSI Center показывал
81%; два ответа battery-query также содержали 81 в байте 8. Байт 9 различает
доступный транспорт:

- `0x02` на `0db0:1620`: беспроводной режим, процент действителен;
- `0x01` на `0db0:b231`: проводной режим и внешнее питание, процент
  действителен;
- `0x01` на `0db0:1620`: клавиатура перешла на кабель, приёмник возвращает
  недействительные 0%.

Декодер проверяет PID вместе с байтом 9. При подключённом кабеле приложение
предпочитает `b231`, поэтому не показывает ложные 0% от приёмника.
После возврата устройства в Linux CLI получил уже 89% и состояние зарядки.
Изменение с 81% подтверждает, что запрос возвращает текущее значение.

## Почему не подходит MSIKLM

Проект [Gibtnix/MSIKLM](https://github.com/Gibtnix/MSIKLM) рассчитан на
ноутбучные клавиатуры SteelSeries с VID/PID `1770:ff00`. Он управляет
подсветкой через 8-байтовый feature-report `0x01`.

У Strike Pro другие HID-устройства (`0db0:1620` и `0db0:b231`), размеры и
report ID. Команды MSIKLM не являются совместимым протоколом. В этом проекте
его код не используется; полезно только независимое подтверждение, что
периферия MSI может управляться через HID feature-report.

## Как найти поле батареи

1. Установить udev-правило, запустить GUI и открыть `Debug` → `Telemetry`.
2. Нажать «Снять снимок» в нескольких известных состояниях: после
   полной зарядки, после заметного разряда, во время зарядки и без кабеля.
3. Экспортировать каждый снимок и сравнить только отчёты одного interface и
   report ID.
4. Отдельно проверить пассивный `0x0d`: разбудить клавиатуру, переключить
   2.4 ГГц и подключить или отключить кабель.
5. Если Linux-снимки не меняются, снять USBPcap-трассу в Windows при запуске
   MSI Center и при обновлении его страницы устройства. Нужна разница между
   двумя состояниями, а не длинная запись обычного набора текста.

Поле подтверждено точным совпадением с 78% в беспроводном режиме, 81% во время
проводной зарядки в MSI Center и последующим значением 89% из Linux CLI.

## Переопределение встроенного профиля

Подтверждённый профиль уже встроен в приложение. Для экспериментов его можно
переопределить: скопировать `config/protocol.example.json` в:

```text
~/.config/kordax/msi-keyboard/protocol.json
```

Поля:

- `source`: `input` или `feature`;
- `interface`: номер USB HID interface;
- `report_id`: первый байт отчёта в десятичном виде;
- `match_prefix_hex`: обязательный префикс ответа для защиты от ложного
  декодирования других vendor-команд;
- `percentage_offset`: индекс байта процента, считая report ID как offset 0;
- `transport_offset`: индекс байта состояния транспорта;
- `wireless_transport_value` и `wired_transport_value`: допустимые значения
  для соответствующих PID;
- `charging_offset` и `charging_mask`: необязательный бит состояния зарядки.

После изменения открыть `Debug` → `Telemetry` и нажать «Перечитать профиль».
